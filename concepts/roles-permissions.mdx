## Overview

VitaFem implements a simple but effective **role-based access control (RBAC)** system to manage user permissions across the healthcare platform. The system distinguishes between three primary user types with distinct capabilities.

## Role System Architecture

### Role Storage

User roles are stored as a simple string field in the `USUARIOS` table:

```sql Oracle Schema
CREATE TABLE USUARIOS (
    ID_USUARIO NUMBER PRIMARY KEY,
    NOMBRE VARCHAR2(255),
    CORREO VARCHAR2(100) UNIQUE,
    PASSWORD VARCHAR2(255),
    ROLE VARCHAR2(20),  -- 'admin', 'doctor', 'patient'
    ACTIVO CHAR(1) DEFAULT '1',
    FOTO_PERFIL VARCHAR2(500)
);
```

<Note>
  While the codebase shows a `role` field in routes and seeders, the User model in `app/Models/User.php` does not explicitly define this field. You may need to add `'ROLE'` to the `$fillable` array to use role-based features.
</Note>

### User Model Extension

To enable role support, update the User model:

```php app/Models/User.php
class User extends Authenticatable
{
    use HasApiTokens, HasFactory, Notifiable;

    protected $table = 'USUARIOS';
    protected $primaryKey = 'ID_USUARIO';
    public $sequence = 'SEQ_USUARIOS';
    public $timestamps = false;

    protected $fillable = [
        'NOMBRE',
        'CORREO',
        'PASSWORD',
        'ROLE',        // Add this field
        'ACTIVO',
        'FOTO_PERFIL',
    ];

    protected $hidden = [
        'PASSWORD',
    ];

    // Helper method to check roles
    public function isAdmin()
    {
        return $this->ROLE === 'admin';
    }

    public function isDoctor()
    {
        return $this->ROLE === 'doctor';
    }

    public function isPatient()
    {
        return $this->ROLE === 'patient';
    }
}
```

## Role Middleware

The `RoleMiddleware` enforces role-based access control on protected routes.

### Middleware Implementation

```php app/Http/Middleware/RoleMiddleware.php
namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;
use Symfony\Component\HttpFoundation\Response;

class RoleMiddleware
{
    /**
     * Handle an incoming request.
     *
     * @param  \Illuminate\Http\Request  $request
     * @param  \Closure  $next
     * @param  string  $role  The required role
     * @return mixed
     */
    public function handle(Request $request, Closure $next, $role)
    {
        $user = $request->user();

        // Check if user is authenticated and has the required role
        if (!$user || $user->role !== $role) {
            return response()->json([
                'message' => 'Acceso denegado. No tienes permisos de administrador.'
            ], Response::HTTP_FORBIDDEN);
        }

        return $next($request);
    }
}
```

### Middleware Registration

The middleware is registered in `app/Http/Kernel.php`:

```php app/Http/Kernel.php
protected $routeMiddleware = [
    'auth' => \App\Http\Middleware\Authenticate::class,
    'role' => \App\Http\Middleware\RoleMiddleware::class,  // Role-based access
    // ... other middleware
];
```

## Role Definitions

### 1. Admin Role

**Role Value**: `admin`

**Capabilities:**
- Manage all system users
- View all appointments across the system
- Create and manage doctor profiles
- Access administrative dashboards
- Modify appointment statuses
- System configuration

**Use Cases:**
- System administrators
- Clinic managers
- IT support staff

<Accordion title="Admin-Specific Routes">
```php routes/api.php
Route::middleware(['auth:sanctum', 'role:admin'])->group(function () {
    // User management
    Route::post('/admin/users', [UserController::class, 'store']);
    Route::put('/users/{id}', [UserController::class, 'update']);
    Route::delete('/users/{id}', [UserController::class, 'destroy']);

    // Appointment management
    Route::get('/admin/citas', [CitaController::class, 'getAllCitas']);
    Route::put('/admin/citas/{id}/estado', [CitaController::class, 'updateEstado']);

    // Doctor management
    Route::post('/admin/medicos', [CitaController::class, 'crearMedico']);
});
```
</Accordion>

### 2. Doctor Role

**Role Value**: `doctor`

**Capabilities:**
- View assigned appointments
- Access patient medical records
- Create and update consultations
- Write prescriptions
- Manage personal schedule
- Update appointment notes

**Use Cases:**
- Physicians
- Medical specialists
- Healthcare providers

<Accordion title="Doctor-Specific Routes">

**Role Value**: `patient` (or `user`)

**Capabilities:**

- View personal appointments
- Update personal profile

- Access consultation records
- Manage personal information

**Use Cases:**
- Registered patients

## Applying Role-Based Protection

### Method 1: Middleware on Routes

Apply role middleware directly to routes:

```php
// Single role requirement
Route::middleware(['auth:sanctum', 'role:admin'])
    ->post('/admin/users', [UserController::class, 'store']);

// Route group with role
Route::middleware(['auth:sanctum', 'role:doctor'])->group(function () {
    Route::get('/medico/citas', [CitaController::class, 'getCitasMedico']);
    Route::put('/medico/citas/{id}/atender', [CitaController::class, 'atenderCita']);
});
```

### Method 2: Controller-Level Authorization

Check roles within controller methods:

```php app/Http/Controllers/CitaController.php
public function getAllCitas(Request $request)
{
    $user = $request->user();

    // Manual role check
    if ($user->ROLE !== 'admin') {
        return response()->json([
            'message' => 'Acceso denegado'
        ], 403);
    }

    // Proceed with admin-only logic
    $citas = Cita::with(['medico', 'paciente'])->get();
    return response()->json($citas);
}
```

### Method 3: Policy-Based Authorization

Create Laravel policies for fine-grained control:

```php app/Policies/CitaPolicy.php
namespace App\Policies;

use App\Models\User;
use App\Models\Cita;

class CitaPolicy
{
    public function viewAny(User $user)
    {
        return $user->ROLE === 'admin';
    }

    public function view(User $user, Cita $cita)
    {
        // Admins can view all
        if ($user->ROLE === 'admin') {
            return true;
        }

        // Doctors can view their appointments
        if ($user->ROLE === 'doctor') {
            $medico = $user->medico;
            return $medico && $cita->ID_MEDICO === $medico->ID_MEDICO;
        }

        // Patients can view their own appointments
        $paciente = $user->paciente;
        return $paciente && $cita->ID_PACIENTE === $paciente->ID_PACIENTE;
    }

    public function update(User $user, Cita $cita)
    {
        return $user->ROLE === 'admin' || 
               ($user->ROLE === 'doctor' && 
                $cita->ID_MEDICO === $user->medico->ID_MEDICO);
    }
}
```

Register the policy:

```php app/Providers/AuthServiceProvider.php
protected $policies = [
    Cita::class => CitaPolicy::class,
];
```

Use in controllers:

```php
public function show(Request $request, $id)
{
    $cita = Cita::findOrFail($id);

    // Check policy
    $this->authorize('view', $cita);

    return response()->json($cita);
}
```

## Default Role Assignment

### Registration with Default Role

```php app/Http/Controllers/AuthController.php
public function register(Request $request)
{
    $request->validate([
        'name' => 'required|string|max:255',
        'email' => 'required|string|email|max:100|unique:USUARIOS,CORREO',
        'password' => 'required|string|min:6',
    ]);

    $user = new User();
    $user->NOMBRE = $request->name;
    $user->CORREO = $request->email;
    $user->PASSWORD = Hash::make($request->password);
    $user->ROLE = 'patient';  // Default role
    $user->ACTIVO = '1';
    $user->save();

    $token = $user->createToken('authToken')->plainTextToken;

    return response()->json([
        'message' => 'Usuario registrado',
        'user' => $user,
        'token' => $token
    ], 201);
}
```

### Social Login Role Assignment

```php routes/api.php
Route::get('/auth/{provider}/callback', function ($provider) {
    $socialUser = Socialite::driver($provider)->stateless()->user();
```

## Role-Based Response Filtering

Customize API responses based on user role:

```php
public function getCitas(Request $request)
{
    $user = $request->user();
    $query = Cita::with(['medico.usuario', 'paciente']);

    // Filter based on role
    switch ($user->ROLE) {
        case 'admin':
            // Admins see all appointments
            break;

        case 'doctor':
            // Doctors see only their appointments
            $medico = $user->medico;
            $query->where('ID_MEDICO', $medico->ID_MEDICO);
            break;

        case 'patient':
            // Patients see only their appointments
            $paciente = $user->paciente;
            $query->where('ID_PACIENTE', $paciente->ID_PACIENTE);
            break;

        default:
            return response()->json(['message' => 'Rol no reconocido'], 403);
    }

    $citas = $query->orderBy('FECHA_HORA', 'desc')->get();
    return response()->json($citas);
}
```

## Security Considerations

<Warning>
  **Important Security Notes:**
  - Never expose role assignment endpoints publicly
  - Only admins should be able to change user roles
  - Validate role values against a whitelist
  - Log all role changes for audit trails
  - Use HTTPS in production to protect tokens
</Warning>

```php Secure Role Update Example
public function updateRole(Request $request, $userId)
{
    // Only admins can change roles
    if ($request->user()->ROLE !== 'admin') {
        return response()->json(['message' => 'Acceso denegado'], 403);
    }

    $request->validate([
        'role' => 'required|in:admin,doctor,patient'
    ]);

    $user = User::findOrFail($userId);
    
    // Log the change
    Log::info('Role changed', [
        'admin_id' => $request->user()->ID_USUARIO,
        'target_user_id' => $userId,
        'old_role' => $user->ROLE,
        'new_role' => $request->role
    ]);

    $user->ROLE = $request->role;
    $user->save();

    return response()->json([
        'message' => 'Rol actualizado',
        'user' => $user
    ]);
}
```
