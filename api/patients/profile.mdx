## Endpoint

<ParamField path="method" type="string" required>
  GET
</ParamField>

<ParamField path="endpoint" type="string" required>
  `/api/pacientes/perfil/{id_usuario}`
</ParamField>

## Authentication

This endpoint is currently **public** and does not require authentication.

<Note>
  In production environments, consider protecting this endpoint with authentication middleware to ensure users can only access their own profiles.
</Note>

## Path Parameters

<ParamField path="id_usuario" type="number" required>
  The user ID from the USUARIOS table to retrieve the profile for
</ParamField>

## Implementation Details

### Oracle Field Handling

This endpoint demonstrates Oracle-specific case handling:

<Warning>
  **Field Name Casing**: The Yajra Oracle driver returns database column names in lowercase when reading from the database, even though they're defined in UPPERCASE in the model. The controller explicitly handles this:
  
  ```php
  // Reading from model (lowercase)
  'NOMBRE_COMPLETO' => $paciente ? $paciente->nombre_completo : $user->nombre,
  'TELEFONO' => $paciente ? $paciente->telefono : '',
  ```
</Warning>

### Controller Method

Implemented in `PacienteController::getProfile()` at `app/Http/Controllers/PacienteController.php:122`

```php
public function getProfile($id_usuario)
{
    // 1. Find user
    $user = User::where('ID_USUARIO', $id_usuario)->first();

    if (!$user) {
        return response()->json(['message' => 'Usuario no encontrado'], 404);
    }

    // 2. Find patient
    $paciente = Paciente::where('ID_USUARIO', $id_usuario)->first();

    // 3. Merge data (reading in lowercase due to Oracle driver)
    return response()->json([
        'ID_USUARIO' => $user->id_usuario,
        'NOMBRE_COMPLETO' => $paciente ? $paciente->nombre_completo : $user->nombre,
        'CORREO' => $user->correo,
        'TELEFONO' => $paciente ? $paciente->telefono : '',
        'FECHA_NACIMIENTO' => $paciente ? $paciente->fecha_nacimiento : '',
        'TIPO_SANGRE' => $paciente ? $paciente->tipo_sangre : '',
        'ALERGIAS_PRINCIPALES' => $paciente ? $paciente->alergias_principales : '',
        'SEXO' => $paciente ? $paciente->sexo : 'F',
        'FOTO_PERFIL' => $user->foto_perfil ? asset('storage/' . $user->foto_perfil) : null
    ]);
}
```

## Data Sources

This endpoint merges data from two tables:

1. **USUARIOS table**: Contains authentication and basic user info
   - `ID_USUARIO`
   - `CORREO`
   - `NOMBRE` (fallback if no patient record)
   - `FOTO_PERFIL` (stored path)

2. **PACIENTES table**: Contains medical and detailed patient info
   - `NOMBRE_COMPLETO` (preferred over user's name)
   - `TELEFONO`
   - `FECHA_NACIMIENTO`
   - `TIPO_SANGRE`
   - `ALERGIAS_PRINCIPALES`
   - `SEXO`

<Info>
  **Graceful Fallbacks**: If no patient record exists for the user, the endpoint returns the user's basic information with empty strings for missing medical data. This allows the profile to be retrieved even for users who haven't completed their medical information.
</Info>

## Profile Photo

The `FOTO_PERFIL` field:

- Stored in the `USUARIOS` table as a relative path (e.g., `avatars/abc123.jpg`)
- Returned as a full URL using Laravel's `asset()` helper
- Files are stored in `storage/app/public/avatars/`
- Returns `null` if no photo has been uploaded

<Tip>
  The profile photo is managed through the update profile endpoint. See the [Update Patient Profile](/api/patients/update-profile) documentation for upload instructions.
</Tip>
