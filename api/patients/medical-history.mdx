## Endpoint

<ParamField path="method" type="string" required>
  GET
</ParamField>

<ParamField path="endpoint" type="string" required>
  `/api/medico/paciente/{id_paciente}/historial`
</ParamField>

## Authentication

This endpoint is currently **public** and does not require authentication.

<Warning>
  In production, this endpoint MUST be protected with authentication and authorization middleware. Only authorized medical personnel should access patient medical histories.
</Warning>

## Path Parameters

<ParamField path="id_paciente" type="number" required>
  The unique patient identifier (ID_PACIENTE) to retrieve medical history for
</ParamField>

## Implementation Details

### Route Definition

Defined in `routes/api.php:157`:

```php
Route::get('/medico/paciente/{id_paciente}/historial', 
    [CitaController::class, 'getHistorialPaciente']);
```

<Note>
  This endpoint is implemented in the `CitaController` (Appointments Controller) rather than `PacienteController` because it primarily deals with appointment history data.
</Note>

### Data Sources

This endpoint queries multiple Oracle tables:

1. **PACIENTES**: Patient demographic and medical information
2. **CITAS**: Appointment records with dates, times, and status
3. **MEDICOS**: Physician information including name and specialty
4. **Related tables**: Any additional appointment details (diagnoses, treatments, notes)

## Use Cases

<Info>
  **Medical Personnel Dashboard**: This endpoint is designed for healthcare providers to view a patient's complete medical history before or during appointments.
</Info>

### Typical Usage Scenarios:

1. **Pre-appointment Review**: Doctor reviews history before seeing patient
2. **Continuity of Care**: New physician reviews previous treatments
3. **Emergency Reference**: Quick access to allergies, blood type, and recent care
4. **Treatment Planning**: Review previous diagnoses to inform current treatment

## Security Considerations

<Warning>
  **HIPAA/Privacy Compliance**: This endpoint exposes sensitive medical information. In production:
  
  - Require authentication with `auth:sanctum` middleware
  - Implement role-based access control (only doctors/nurses)
  - Log all access for audit trails
  - Verify the requesting user has permission to view this specific patient
  - Consider implementing patient consent tracking
</Warning>

```php
// Recommended protection:
Route::middleware(['auth:sanctum', 'role:medico,admin'])->group(function () {
    Route::get('/medico/paciente/{id_paciente}/historial', 
        [CitaController::class, 'getHistorialPaciente']);
});
```

## Chronological Ordering

Appointments in the `historial` array are typically ordered by:

1. **Date** (FECHA): Most recent first, or oldest first depending on implementation
2. **Time** (HORA): When multiple appointments on same date

<Tip>
  Verify the sort order in your implementation. Medical histories are often displayed in reverse chronological order (newest first) for quick access to recent care.
</Tip>

## Oracle-Specific Considerations

<Info>
  **Field Casing**: As with other patient endpoints, be aware that the Yajra Oracle driver may return column names in lowercase even though they're defined in UPPERCASE in the model.
</Info>
